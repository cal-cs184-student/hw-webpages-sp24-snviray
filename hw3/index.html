Explain the triangle intersection algorithm you implemented in your own words.

The triangle intersection algorithm follows from the following equations

  • r(t) = O + tD (ray equation)
  • P = P_0α +P_1β + P_2γ (barycentric coordinates)

Since α + β + γ = 1, we can rewrite

  • α = (1 - b_1 - b_2)
  • β = b_1
  • γ = b_2

By substituting for α, β, and γ and setting the ray and barycentric coordinate equations equal, we get.

O − P_0 = −tD + b_1(P_1 − P_0) + b_2(P_2 − P_0)

The matrix form of this equation can be solved to obtain b_1, b_2, and t values. It uses Cramer’s rule, which solves linear equations using matrix determinants.

Therefore, solving for the determinants of the matrices and then plugging them into Cramer’s rule allows for solving the equation to get t for the intersection.

To implement triangle intersection test, I first implemented the algorithm described above to obtain b_1 , b_2 , and t values. To test for intersection, I checked that b_2 and b_1 were in between 0 and 1 and 1 - b1 - b2 was also between 0 and 1. The reason why I must do these checks is to make sure that the intersection is within the triangle. If α + β + γ = 1 , then the intersection point can be represented in barycentric coordinates and therefore must be in the triangle.

If the test passes,

  • I set the ray’s max_t to be the smallest t value if it was smaller than the current max_t. I only do the following if this is true
  • I changed the Intersection struct values to the following
    • t was the t value .
    • n was the barycentric interpolation of the three normals of each point on the triangle. alpha * n1 + beta * n2 + gamma * n3
    • primitive was this (the sphere itself)
    • bsdf was just the return value of get_bsdf

Part 2

Walk through your BVH construction algorithm. Explain the heuristic you chose for picking the splitting point.

  1. I computed a bounding box for all the primitives passed into the function, calculating the length and summing the centroid positions of each primitive’s bounding box while doing so
  1. I divided the sum of the centroids by the length to get the average centroid
  1. I initialized the BVHNode with the overall bounding box
  1. If the amount of primitives passed in are less than max_leaf_size, I set the left and right nodes to NULL and set the passed in start and end to the node’s start and end
  1. Otherwise, I divided the primitives into six left and right collections (vectors of primitives) about each axis (x, y, and z) depending on whether each centroid was above or below the average on that axis.
    • While doing so, I also computed the amount of primitives I pushed to the left as x_count, y_count, and z_count
    • I ran this check for each set of right and left collections to find the most evenly distributed axis to divide upon. x_check = abs(x_count - length / 2)
    • I used the axis with the smallest check, as long as both collections weren’t empty for that axis
    • If one of the collections for the selected axis was empty, I just used an axis that had two nonempty collections
    • If none of the axes had none nonempty collections, I split up the primitives randomly.
  1. After splitting up the primitives, I ran construct_bvh recursively on the left and right collections, setting the current node’s left and right as their results.
  1. I returned the BVHNode.

Compare rendering times on a few scenes with moderately complex geometries with and without BVH acceleration. Present your results in a one-paragraph analysis.

Without BVH acceleration, My maxplanck took 24.6102s to render and CBLucy took 111.2615s. When I added bvh, maxplanck took 0.2642s and CBLucy took 0.2903s. maxplanck sped up by 93.15x and CBLucy sped up by 383.26x.
